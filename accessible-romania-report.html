Accessibility overlay

An accessibility overlay was found on each page of the website. This tool doesn't provide full compliance with accessibility standards and laws and can impact the overall accessibly in a negative way.

Upon a quick look, this overlay introduces the following issues:

  • Introduces many tab stops for keyboard users and screen reader users, without providing a way to bypass this repeated content
  • Wrongfully uses links instead of buttons for some of its functionalities. These features are then announced as links to screen reader users and can cause confusion.

Overlays pose many issues in general:

  • Read more about accessibility overlays in English
  • Read more about accessibility overlays in Romanian

Recommendation: Remove the overlay widget.

Additional evaluation requirements

This report will include a list of issues and warnings per each Success Criterion:

  • Issues represent violations of WCAG (Web Content Accessibly Guidelines).
  • Warnings are not direct violations of WCAG, but could still impact a user's experience in a negative way.

For each issue or warning, the report will contain: how the issue was found, how it might impact users and how it could be solved.

Important: In case of multiple violations of the same Success Criteria on multiple pages, the issue will be presented in a generic way, only showing 2-3 examples of the violation (instead of repeating the same issue for each page).

To save time while auditing, I would like to delegate the task of finding identical issues to the ones in this report to the development team that will also handle the remediation of the website.

1.1 Text Alternatives
Success Criterion Result Observations
1.1.1: Non-text Content(opens new tab)

Result: Not checked

Observations:

Issue: The language switcher doesn't provide descriptive enough alt text. "en" is easy to miss or misunderstand.

Use the full word for languages, e.g. "English".

Warning: The alt text on the logo image is redundant and will not be spoken to screen reader users.

This happens because the image lives inside a link parent element that uses aria-label to provide an accessible name instead.

Mark the image as decorative (by using empty alt text).

2.2.2: Pause, Stop, Hide(opens new tab)

Result: Failed

Observations:

Issue: The auto playing slideshow found after the Testimonials section cannot be stopped or paused and might cause distractions for users with attention deficits.

Though there's a mechanism to stop the slideshow animation for mouse users (on mouse hover), keyboard only users cannot stop the animation.

Implement a "pause", "stop" or "hide slideshow" button for this slideshow.

2.4 Navigable
Success Criterion Result Observations
2.4.1: Bypass Blocks(opens new tab)

Result: Failed

Observations:

Issue: The skip link at the beginning of the page doesn't work because it doesn't bypass repeated content. Keyboard only users can't skip over repeated content such as the main menu.

Upon pressing the Enter key, this link should move the user's focus to the content of the page, ideally content that starts with the main title of the content (h1). Focus on the new content should also be visible - 2.4.7: Focus Visible.

2.4.7: Focus Visible(opens new tab)

Result: Failed

Observations:

Issue: Not all elements receive a visible focus indicator. Keyboard only users can easily get lost if they don't know what element they're currently focused on.

Examples:

  • The logos and the left and right arrow buttons that control the slideshow that follows the Testimonials section
  • The images in the footer of the page

Make sure the focus outline is visible against the element's background, ideally with a contrast ratio of at least 3 to 1.
